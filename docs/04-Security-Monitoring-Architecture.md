# Security Monitoring Architecture

## Introduction
This stage of the project focuses on designing a security monitoring architecture capable of improving cyberattack visibility within a simulated Small and Medium sized Enterprise (SME). In many SME environments, valuable security information is generated by endpoints, servers and network devices. However these logs often remain isolated on individual systems, making it difficult to reconstruct incidents, identify attacker behaviour and respond effectively to security events. To address this challenge, this project adopts a centralised monitoring approach where host and network telemetry will be collected, analysed and correlated to provide greater visibility across the environment. Rather than investigating each system independently, security events from Windows, Linux and network traffic will be combined to create a more complete understanding of system activity.
This architecture forms the foundation for the remaining chapters of the project, where telemetry collection, log correlation and security investigations will be implemented.

## Security Monitoring Objectives

The monitoring architecture has been designed to achieve the following objectives:

- Improve visibility across Windows and Linux systems.
- Collect host and network telemetry from multiple sources.
- Correlate security events originating from different systems.
- Detect suspicious administrative and authentication activity.
- Reconstruct attack timelines using multiple evidence sources.
- Demonstrate how SMEs can improve security monitoring using open source technologies.

## Telemetry Sources

Each VM within the laboratory contributes different types of security telemetry. Individually these logs provide valuable information but when we correlate them they offer significantly greater visibility into system activity.

| System | Telemetry Source | Information Collected | Investigation Purpose |
|---------|------------------|----------------------|-----------------------|
| Windows 11 Workstation | Windows Security Log | User authentication and security events | User activity monitoring |
| Windows 11 Workstation | PowerShell Operational Log | Administrative command execution | PowerShell behavioural analysis |
| Ubuntu Server | auth.log | SSH authentication attempts | Login monitoring |
| Ubuntu Server | journalctl | System and service events | System behaviour monitoring |
| Ubuntu Server | UFW Firewall | Firewall activity | Network security monitoring |
| Ubuntu Desktop | Wireshark Packet Captures | Network communication | Traffic analysis and correlation |

## Security Monitoring Architecture

The laboratory consists of three VMs connected through a private virtual network. Each system performs a different role within the monitoring architecture:

- **Windows 11 Workstation** represents a typical employee endpoint where administrative activity, authentication events and PowerShell execution occur.
- **Ubuntu Server** represents a production server hosting Linux services while also acting as the **central security monitoring server.**
- **Ubuntu Desktop** functions as the security analyst workstation used to administer systems, capture network traffic and investigate alerts.

To improve visibility across the environment, security telemetry generated by Windows and Ubuntu will be forwarded to a central monitoring platform. Rather than analysing logs separately on each machine events will be collected and correlated to provide a unified view of system activity.
The following architecture will be implemented during the remaining chapters of the project:

```text
                    Windows 11 Workstation
               Security • PowerShell • Authentication
                              │
                         Wazuh Agent
                              │
                              ▼
                     Ubuntu Server
                 Wazuh Manager & Correlation
                 Linux Logs • SSH • Firewall
                              │
                              ▼
                    Wazuh Dashboard
                              │
                              ▼
                Ubuntu Desktop Analyst Workstation
           Wireshark • Investigation • Incident Analysis
```



## Correlation Strategy

The primary objective of this project is to correlate events generated across multiple systems and security investigations will combine evidence from each telemetry source to reconstruct attacker activity and improve cyberattack visibility.

For example, an SSH connection initiated from the Windows workstation may generate:

- A PowerShell event on Windows.
- An authentication event within the Ubuntu Server.
- Network traffic captured using Wireshark.
- A correlated alert within Wazuh.

Individually these events provide only partial visibility. When combined, they allow an investigator to understand the sequence of events, identify the affected systems and reconstruct the complete attack timeline.

## Design Decisions

Several design decisions were made during the development of this laboratory.

| Design Decision | Justification |
|-----------------|---------------|
| Three virtual machines | Simulates a realistic SME environment while remaining manageable on a single host computer. |
| Windows endpoint | Represents employee workstation activity. |
| Ubuntu Server | Represents production server infrastructure and hosts the monitoring platform. |
| Ubuntu Desktop | Dedicated analyst workstation for monitoring and investigation. |
| Wazuh | Selected as an open source SIEM suitable for SME environments. |
| Wireshark | Provides packet level visibility to complement host based telemetry. |

## Current Limitations

The laboratory is intentionally designed as a small scale environment and therefore has several limitations.

- The environment contains three virtual machines rather than a production enterprise network.
- Sysmon is unavailable on the Windows ARM virtual machine, limiting process creation telemetry.
- Encrypted protocols such as SSH prevent inspection of application payloads using packet captures alone.
- Alert quality depends on the logging configuration and correlation rules implemented within Wazuh.

# Summary

This chapter established the security monitoring architecture that will be implemented throughout the remainder of the project.

Rather than treating Windows, Linux and network monitoring as separate activities, the proposed design integrates host and network telemetry into a centralised monitoring platform capable of correlating security events across multiple systems.

The next chapter focuses on implementing this architecture by deploying Wazuh, configuring log collection and preparing the environment for multi source event correlation and security investigations.
